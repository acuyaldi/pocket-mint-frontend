# v0.3.0-rc.2 — Staging Validation

Tanggal validasi: **18 Juli 2026**. Dijalankan sebagai sesi terpisah dari
`mvp-stable-rc-validation.md` §1–§17 (audit berbasis kode/test lokal), fokus
khusus pada **migration dan smoke test terhadap database yang benar-benar
disediakan (bukan hanya `npx vitest run`)**, sesuai keputusan produk baru di
bawah.

## 0. Keputusan lingkup: tidak ada environment "staging" terpisah

Sebelum sesi ini dimulai, dikonfirmasi ke pemilik produk: **Pocket Mint
adalah proyek pribadi tanpa environment staging terpisah.** Hanya ada dua
environment: **local/dev** dan **production**. Ini menggantikan asumsi lama
di `PM-STAB-003`/`PM-STAB-004` yang mengharapkan deployment Railway/Supabase
"staging" terpisah (`AGENTS.md`/`agent-rules.skill.md` backend menyebut `dev`
branch → "Railway Staging" — deployment ini **tidak pernah benar-benar
dibuat**, dikonfirmasi ulang tidak ada config Railway/Vercel/Fly/Docker di
kedua repo).

**Keputusan yang dipakai untuk sesi ini:** "staging" dalam task divalidasi
memakai **database PostgreSQL disposable lokal yang benar-benar berjalan**
(bukan mock, bukan hanya unit test) sebagai pengganti staging terpisah —
migration, aplikasi, dan smoke test seluruhnya nyata, hanya databasenya
bersifat sementara/lokal alih-alih database cloud terpisah. Database
production (`.env` `DATABASE_URL`/Supabase) **tidak pernah disentuh** pada
sesi ini — tidak dibaca datanya, tidak dijalankan migration apa pun
terhadapnya.

## 5. Backup/restore staging

1. **Backup setelah data smoke test tersedia:** `npm run db:backup`
   (`BACKUP_SOURCE_URL` → database disposable berisi data smoke test §4) →
   sukses, `.dump` custom-format, 0.1 detik, ~0.02 MiB.
2. **Restore ke database staging disposable baru:** instance PostgreSQL
   disposable kedua di-provision (port 55778, database kosong baru).
   `npm run db:restore` (`CONFIRM_RESTORE=yes`) → sukses, 0.1 detik.
3. **Guard restore diuji ulang (ketiganya benar menolak):**
   - Tanpa `CONFIRM_RESTORE` → ditolak, exit 1.
   - Host menyerupai Supabase (`db.abcdefgh.supabase.co`) → ditolak, exit 1.
   - Restore ke database yang sudah berisi tabel tanpa `--force` → ditolak,
     exit 1 ("already has 6 table(s)... Refusing to restore over existing
     data").
4. **Verifikasi schema dan data:** `npm run db:verify` pada source **dan**
   target:

   ```
   Row counts (kedua database, identik persis):
     users: 2, wallets: 3, transactions: 6, installments: 1, categories: 24
   FK check — 0 orphaned row(s) di seluruh 4 relasi yang diperiksa
   Foreign key constraints present: 9
   VERIFY OK (kedua database)
   ```

5. **Read-only smoke test pada database hasil restore:** backend baru
   di-boot melawan database hasil restore (port 5597).
   `GET /health` → `200`. `GET /v1/dashboard/summary` dengan token user

   PD-001 benar dan berasal dari data hasil restore, bukan data asli.
   `GET /v1/wallets` → `200`, kedua wallet user tersebut tampil dengan
   saldo yang benar.

**Status: PASS** — backup, restore, verifikasi row count/FK, dan smoke

## 9. Catatan teknis (untuk validasi berikutnya)

- **Trap dotenv:** proses backend memuat `dotenv/config` yang **tidak
  menimpa** environment variable yang sudah di-set di proses induk, tapi
  **akan** mengisi variabel yang belum di-set dari `.env` lokal. Saat
  menjalankan backend melawan database disposable dengan secret JWT
  test-only, `SUPABASE_URL` harus di-set eksplisit ke string kosong (bukan
  dibiarkan unset) — jika tidak, `.env` production akan diam-diam mengisi
  `SUPABASE_URL` asli dan verifikasi JWT beralih ke JWKS Supabase nyata
  (menolak token test-only dengan `401` yang membingungkan, terlihat
  seperti bug auth padahal konfigurasi environment). Ini murni isu
  prosedur validasi, bukan bug aplikasi — didokumentasikan di sini agar
  sesi validasi berikutnya tidak mengulang jebakan yang sama.
- Seluruh file skrip sementara (`tmp-*.mjs`, `tmp-backups/`) dibuat khusus
  untuk sesi ini di working tree `pocket-mint-be` dan **dihapus di akhir
  sesi** — tidak ada yang di-commit, `git status` kedua repo bersih
  sebelum dan sesudah sesi ini.

  kredensial Supabase (`PM-STAB-003`) dan reconciliation migration

  --applied` terhadap database Supabase nyata yang dipakai aplikasi
  sehari-hari — beda dari "provisioning database kosong" yang sudah PASS
  di sesi ini, lihat §11 `known-issues.md` PM-STAB-004). Keduanya
  **tidak dijalankan sengaja** pada sesi ini karena menyentuh database
  production sungguhan membutuhkan persetujuan eksplisit terpisah dan
  jendela backup — bukan sesuatu yang aman dieksekusi otomatis oleh agent
  ini tanpa konfirmasi langsung dari pemilik produk.
- Tidak ditemukan regresi Critical/High baru pada sesi ini.

**Bukan Fail / Rollback Required** — tidak ada kegagalan migration,
kegagalan smoke test, atau temuan finansial-integritas yang salah pada
sesi ini.

## Addendum — 19 Juli 2026: PM-STAB-003 resolved after forensic verification

§10 dan §11 di atas mencatat `PM-STAB-003` (rotasi kredensial Supabase)
sebagai Open/non-blocking-untuk-sesi-ini **pada titik waktu sesi ini
berlangsung** — entri di atas tidak diubah, untuk menjaga jejak audit

full-history atas kedua repo (`pocket-mint-be`, `pocket-mint-fe`)
mengonfirmasi tidak ada kredensial database privileged
(`DATABASE_URL`/`DIRECT_URL`, password DB, service-role key) yang pernah
ter-commit, dan project Production (`wvrdnmiuyeecqatlwbpp`) tidak pernah
muncul di git history manapun. `PM-STAB-003` diperbarui menjadi **Resolved
after forensic verification** di `known-issues.md` dan `release-status.md`
— bukan karena rotasi dieksekusi, melainkan karena audit forensik
mengonfirmasi tidak ada kredensial privileged yang perlu dirotasi. Detail
lengkap: `known-issues.md` PM-STAB-003. `PM-STAB-004` tidak terpengaruh
oleh addendum ini dan tetap Open (lihat §11 di atas).
